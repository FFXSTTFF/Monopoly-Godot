# Дизайн: светлая тема + камера от лица игрока за столом

Дата: 2026-07-22
Статус: утверждён по итогам визуального брейнстрима, готов к планированию

## Цель

Два связанных, но независимых изменения, оба применяются ко **всей игре**
(меню, лобби, HUD, доска — не только к паку `megapolis`):

1. Перекрасить визуальную тему из тёмной "казино-люкс" в светлую,
   сохранив существующую структуру карточек/панелей/кнопок.
2. Заменить единственную общую ортографическую камеру на перспективную,
   стартующую у места локального игрока за столом, с горизонтальным
   вращением по перетаскиванию (без свободного перемещения, без зума).

Обе фичи не трогают сетевой протокол, `core/`, паки или контент — это
чисто `render/`/`ui/` слой.

## 1. Светлая тема

### Подход: перекрасить `UiPalette`, не переписывать потребителей

`UiPalette.gd` — единственный источник цветов для `LuxuryTheme` (2D UI:
кнопки, панели, поля ввода) и для большинства ручной раскраски в `Hud.gd`,
`MainMenu.gd`, `WaitingRoom.gd`, `SettingsScreen.gd`, `TradeDialog.gd`,
`AssetDialog.gd`, `TokenCustomizer.gd`. Все они уже обращаются к цветам по
именованным константам (`UiPalette.GOLD`, `UiPalette.INK`, ...), а не по
хардкоженным hex-значениям. Поэтому самое эффективное изменение — заменить
**значения** констант в `UiPalette.gd` на светлый набор, оставив **имена**
как есть (переименовывать 40+ мест использования не нужно и не даёт
никакой выгоды для однопользовательского хобби-проекта).

Ориентировочная светлая палитра (заменяет текущую тёмную):

| Константа | Было (тёмное) | Станет (светлое) |
|---|---|---|
| `INK` (фон/базовый) | `#090d0b` | `#eef1f5` |
| `FELT_DARK` | `#071d14` | `#dfe4ea` |
| `FELT` | `#0d3a27` | `#f3f5f8` |
| `FELT_LIGHT` | `#18563b` | `#ffffff` |
| `MAHOGANY` / `MAHOGANY_LIGHT` (акцент кнопок) | `#2b120d` / `#4a2116` | `#c9762f` / `#e0a05a` |
| `GOLD_DARK` / `GOLD` / `GOLD_LIGHT` (рамки/акцент) | тёмно-золотые | `#b8933a` / `#c9a54a` / `#e6cf8f` (оставляем золото как акцент, просто чуть менее контрастное на светлом фоне) |
| `IVORY` (текст) | `#f5efd9` | `#1e2430` |
| `MUTED` (второстепенный текст) | `#b5aa8d` | `#6c7688` |
| `DANGER` / `SUCCESS` | без изменений | без изменений (уже достаточно контрастны) |
| `GLASS` (полупрозрачная подложка панелей) | тёмное `rgba(0.025,0.055,0.042,0.92)` | светлое `rgba(1,1,1,0.92)` |

Точные значения уточняются на этапе реализации по месту (важен принцип
"поменять значения, не трогать структуру"), утверждённый ориентир — из
макета в брейнсторме (карточка KURT: белый фон `#ffffff`, рамка
`#d8dce2`, текст `#1e2430`, акцент группы поверх — как сейчас через
`cell.color`).

### 3D-окружение и материалы стола/фетра

`UiPalette` красит только 2D UI. Отдельно нужно осветлить 3D-сцену:

- `render/GameScene.gd` `_setup_environment()`: `background_color` (сейчас
  `#050806`), `ambient_light_color` (сейчас `#385346`), `fog_light_color` —
  заменить на светлые эквиваленты.
- `render/shaders/felt.gdshader` и `render/shaders/wood.gdshader` **не
  требуют правки самого шейдера** — оба уже объявляют цвета как
  `uniform vec3 ... : source_color` (`base_color`/`fiber_color` для фетра,
  `dark_wood`/`light_wood` для дерева) с дефолтами, зашитыми в сам шейдер.
  Сейчас `BoardRenderer`/`GameScene` создают `ShaderMaterial` и назначают
  шейдер, не переопределяя эти uniform'ы. Осветление — это добавление
  `material.set_shader_parameter("base_color", <светлый цвет>)` и т.д. в
  местах создания материала (`_setup_table`/`_build_foundation`), без
  изменения `.gdshader`-файлов.
- `gold.gdshader` не осматривался в деталях — на этапе реализации проверить
  аналогично (вероятно тоже просто `uniform`, золото остаётся золотом на
  светлом фоне, трогать не обязательно).

### Карточка поля (`BoardRenderer._build_property_card`/`_add_card_row`)

Сохраняется структура (см. брейнсторм): лого-квадрат сверху → строка
названия → строка цены → отдельные строки аренды по уровням владения.
Меняются только цвета `StandardMaterial3D.albedo_color` в
`_build_property_card`/`_add_card_row` (сейчас `Color("#2a2a2a")` рамка +
`Color("#d9d9d9")` заливка → на светлые: например рамка `#e2e6ec`, заливка
`#f3f5f8`, текст лейблов `#1e2430` вместо `#1a1a1a`).

### Размер карточек — заметно крупнее

`BoardRenderer.TILE_SIZE` (сейчас `1.6`) и производные пропорции
(`photo_size`, `row_depth`, `CARD_MARGIN`) увеличиваются, чтобы длинные
названия компаний (`Davison Standard Oil Company`, `CINEBERRY Pictures`)
помещались в одну строку без переноса. Точный коэффициент подбирается на
этапе реализации визуальной проверкой в редакторе (нет точного числа,
утверждённого заранее) — ориентир "заметно крупнее", не "чуть больше".
Рост `TILE_SIZE` автоматически растягивает доску/стол/камеру, так как
`GameScene`/`BoardRenderer` уже считают размеры от этой константы
(`board_world_size`, `_table_size`, `_fit_camera`) — отдельно ничего
пересчитывать вручную не нужно.

## 2. Камера от лица игрока за столом

### Проекция

`Camera3D.PROJECTION_PERSPECTIVE` вместо текущего
`PROJECTION_ORTHOGONAL` (`render/GameScene.gd` `_setup_camera()`). FOV и
дистанция подбираются так, чтобы вся доска помещалась в кадр с ближнего
края стола (см. ниже).

### Стартовая позиция — место локального игрока

Каждый клиент независимо вычисляет свою камеру по **своему** месту за
столом. "Место" уже существует как концепция — `PlayerState.order`
(порядковый номер, присваивается в `GameController.add_player`) — то же
самое поле, которое `BoardRenderer._token_world_pos` использует для
раскладки фигурок вокруг клетки. Камера должна использовать ту же логику,
но в масштабе всего стола: местоположение по кругу вокруг доски,
пропорциональное `order` игрока и общему числу игроков за столом
(`NetworkManager.local_players.size()`), направлена в центр доски.

Это read-only, чисто локальное отображение — не сетевая фича: **не**
влияет на `public_snapshot`/`private_state`, не требует изменений в
`core/` или `autoload/NetworkManager.gd`. Каждый клиент вычисляет и хранит
свою камеру независимо, ничего не реплицируется.

### Управление — только горизонтальное вращение

Перетаскивание (мышь/тач) вращает камеру вокруг **вертикальной оси**,
проходящей через центр стола, на **фиксированных высоте и радиусе**
(тех же, что и стартовая позиция игрока). Явно **исключено**: зум,
свободное перемещение (полёт), наклон/вертикальное вращение, "хождение"
игрока. Отпускание перетаскивания оставляет камеру там, где её повернули
(не пружинит обратно к месту игрока) — если нужно вернуть камеру на
место, это отдельная опция, не входит в эту итерацию (см. §3).

### Реализация — новый компонент, не встроенный код в `GameScene.gd`

Учитывая, что `GameScene.gd` уже занимается освещением/окружением/столом/
HUD-слоем, логику камеры стоит вынести в отдельный файл
`render/TableCamera.gd` (`class_name TableCamera`, `extends Camera3D`),
который `GameScene` создаёт и добавляет вместо текущего инлайн-кода
`_setup_camera()`/`_fit_camera()`. `TableCamera` инкапсулирует: расчёт
стартового угла по месту игрока, обработку перетаскивания
(`_unhandled_input`/`_gui_input` на невидимый full-rect `Control` поверх
сцены, либо прямой `_unhandled_input` на самой камере — уточняется в
плане реализации), и пересчёт позиции по углу/радиусу/высоте при ресайзе
окна (переиспользуя `_fit_camera`-подобную логику для радиуса/FOV, но не
для позиции — позиция теперь всегда "по кругу", не "сверху").

## 3. Не входит в эту итерацию

- Возврат камеры "домой" по кнопке/после хода — не запрошено, можно
  добавить позже без изменений в остальной архитектуре.
- Зум/наклон/свободный полёт камеры.
- Раздельные позиции камеры на мобильных/тач-устройствах (проект не
  таргетирует мобильные, не актуально).
- Изменение `gold.gdshader` (не осматривался; вероятно не требует правок,
  проверяется по факту в реализации).
- Точные числовые коэффициенты (величина `TILE_SIZE`, точные hex светлой
  палитры, FOV/радиус камеры) — фиксируются в процессе реализации по
  визуальной проверке, а не заранее в этом документе.

## 4. Проверка

- Визуальная: запустить `solo_test` до и после — сравнить с макетами из
  брейнсторм-сессии (светлая карточка KURT, светлая доска).
  С 2+ локальными наблюдателями (или последовательным подключением второго
  клиента) проверить, что каждый видит камеру именно у **своего** места,
  а не у чужого.
- Перетаскивание камеры не должно быть доступно/влиять на игру, когда
  курсор наведён на HUD-кнопки (не перехватывать клики по UI как вращение
  камеры).
- Регрессия: `classic` ruleset должен выглядеть так же структурно (та же
  карточка, тот же принцип камеры) — светлая тема и новая камера
  применяются глобально, не только к `megapolis`.
