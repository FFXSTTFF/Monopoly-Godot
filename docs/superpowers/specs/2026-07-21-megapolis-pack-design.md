# Дизайн: пак `megapolis` — тема «Мегаполия»

Дата: 2026-07-21
Статус: утверждён, готов к планированию реализации

## Цель

Добавить в игру новый выбираемый ruleset — тематическую реплику настольной
игры «Мегаполия Premium» (бизнес-компании вместо классических цветных улиц),
с собственными механиками особых клеток, не существующими в `classic`.
Реализовано как отдельный **пак** (`packs/megapolis`), зависящий от `core`,
не трогающий `classic.json` и не ломающий текущий геймплей.

## 1. Структура пака

```
packs/megapolis/
  pack.json                     (id: megapolis, dependencies: ["core"], api_version: 2)
  content/
    cells/business_cells.json   (24 property-клетки + 4 новых угла)
    cards/chance.json           (свои карты "Шанс", deck: "chance")
    cards/money.json            (свои карты "Деньги", deck: "treasury")
    rulesets/business.json      (ruleset "megapolis:business")
  scripts/main.gd               (entry-скрипт пака)
```

Токены, роли, звуки — переиспользуются из `core` через зависимость
(`dependencies: ["core"]`), отдельно не определяются.

## 2. Раскладка доски

Без железных дорог и коммуналок. 8 групп × 3 компании = 24 покупаемых поля +
4 угла + 12 спецклеток (по 1 "Деньги"/"Шанс"/налог-проходная на сторону) =
**40 клеток → `board_size: 11`**.

Пары групп по сторонам (сторона = 10 клеток: угол + 9):

| Сторона (после угла) | Группа A | Группа B |
|---|---|---|
| после СТАРТ | Легка промисловість | Телеканали |
| после НАЛОГОВОЙ | Будівельні компанії | Кіностудії |
| после ОТПУСКА | Комп'ютерні компанії | Фінансові компанії |
| после БИРЖИ | Авіакомпанії | Нафтові компанії |

Шаблон стороны: `угол, A1, ДЕНЬГИ, A2, [спец], A3, B1, ШАНС, B2, B3`, где
`[спец]` на трёх из четырёх сторон — обычная промежуточная клетка того же
типа "Деньги"/"Шанс" (не отдельный тип "налог" — см. раздел 4, старая
фиксированная клетка НАЛОГ на этой доске не используется вообще).

**Формула ренты** (мимикрия под `classic.json`, где `rent[0] ≈ price × 0.13`):
`rent_table = [r, r×5, r×15, r×45, r×80, r×112]`, где `r = round(price × 0.13, до 1000)`.
Пример (`brown_1`-эквивалент, price 60 000): `[8000, 40000, 120000, 360000,
640000, 900000]` — совпадает с реальным `brown_1` из `classic.json`.
`mortgage = price / 2`, `rent_growth: 1.1` — как в `classic`. Точные 24×6
чисел генерируются по формуле на этапе реализации, не перечисляются вручную
в этом документе.

`sale_price`/mortgage — по умолчанию движка (не переопределяются).

## 4. Углы и спецклетки — новая механика

Никакой тюрьмы в этом ruleset'е нет вообще: `PlayerState.in_jail` /
`jail_turns` / `get_out_cards` не используются (остаются в движке только для
`classic`).

| Клетка | `type` (новый) | Поведение |
|---|---|---|
| **СТАРТ** | `corner_start` (существующий) | Без изменений — `pass_start_reward` |
| **НАЛОГОВАЯ** | `tax_office` (новый) | `tax_per_business: 20000` — при **остановке** и при **проезде мимо** во время хода списывается `20000 × количество полей, которыми владеет игрок` (все property-поля игрока на доске, не только в этой группе) |
| **ОТПУСК** | `vacation` (новый) | При остановке: следующий ход этого игрока целиком пропускается (флаг `skip_next_turn`), без денег и без выбора действий |
| **БИРЖА** | `exchange` (новый) | При остановке: игрок жмёт "Бросить" вручную до 3 раз; если среди 3 бросков есть хотя бы 2 одинаковых — выигрыш **100 000**, иначе — проигрыш **100 000**. Каждый бросок анимируется как обычный бросок кубика (переиспользуется `EventBus.dice_rolled`, уже отрисовывается `BoardRenderer`) |
| **ШАНС** | `chance` (существующий тип) | Свои карты в `content/cards/chance.json`, без эффектов `send_to_jail`/`get_out_of_jail` |
| **ДЕНЬГИ** | `treasury` (существующий тип, только сменено отображаемое имя) | Свои карты в `content/cards/money.json`, полностью заменяет и КАЗНУ, и фиксированный НАЛОГ classic'а |

## 5. Изменения движка (не только контент)

Общие для движка изменения — не хардкод под конкретный пак, любой будущий
ruleset сможет использовать те же три типа клеток.

### `core/GameController.gd`
- `resolve_roll`: после вычисления `from_index`/`to_index`, проход по всем
  клеткам, **пройденным** между ними (исключая финальную — она уже
  обрабатывается `_resolve_landing`), проверка `type == "tax_office"` для
  налога-на-проезде. Может задеть максимум одну такую клетку за ход (кубик
  1-6 на 40-клеточной доске не даёт полный круг).
- `_resolve_landing`: новые ветки `match`:
  - `tax_office` → тот же налог-расчёт, что и для проезда, затем `_finish_turn`;
  - `vacation` → `player.skip_next_turn = true`, затем `_finish_turn`;
  - `exchange` → `turn_phase = TURN_AWAITING_EXCHANGE`, инициализация
    `pending_exchange = {"peer": peer_id, "rolls": []}`.
- `_begin_current_turn`: если у текущего игрока `skip_next_turn == true` —
  сбросить флаг и сразу передать ход дальше (тот же паттерн, что уже
  используется для пропуска обанкротившихся игроков в `_finish_turn`).
- Новый метод `request_exchange_roll(peer_id) -> Dictionary`: один бросок за
  вызов (валиден только в `TURN_AWAITING_EXCHANGE` и только для текущего
  игрока), добавляет в `pending_exchange.rolls`; после 3-го броска —
  серверное решение (2 из 3 совпали → кредит 100000, иначе — дебет 100000,
  через существующий `_charge`), сброс `pending_exchange`, `_finish_turn`.

### `core/GameRules.gd`
- Новая статическая функция: `portfolio_tax(peer_id, properties, tax_per_business) -> int`
  — считает количество полей текущего владения игрока и умножает на ставку
  из cell-def.

### `core/PlayerState.gd`
- Новое поле `skip_next_turn: bool = false`, добавляется в
  `to_public_dict`/`from_public_dict` (публичное — другие игроки должны
  видеть, что кто-то пропускает ход).

### `autoload/NetworkManager.gd`
- Новая пара `request_exchange_roll()` / `@rpc server_request_exchange_roll()`,
  зеркально `request_roll`/`server_request_roll`. Результат каждого броска
  идёт через уже существующий `EVENT_ROLL`-канал (переиспользуется
  `EventBus.dice_rolled`), финальный исход (выигрыш/проигрыш) — через
  `EVENT_ACTION` с новым именем `"exchange_resolved"`.

### `ui/Hud.gd`
- Новая ветка в `_refresh()` для `turn_phase == "awaiting_exchange"`:
  контекст-панель показывает уже выпавшие броски (например "Броски: 4, 2 —
  бросьте ещё раз") и кнопку "БРОСИТЬ", активную пока бросков меньше 3.
  Использует существующий `_add_action`, без нового диалога/сцены —
  анимация уже приходит бесплатно через `BoardRenderer`, который слушает
  `dice_rolled` независимо от HUD.
- Новый case в `_on_game_action` для тоста `"exchange_resolved"`
  ("Биржа: выигрыш 100 000 $" / "Биржа: проигрыш 100 000 $").

## 6. Контент

`content/cards/chance.json` и `content/cards/money.json` — новые тексты под
бизнес-тематику (аудит, дивиденды, проверка ФНС, премия инвестора и т.п.),
используют только эффекты `credit`/`debit`/`move_to`/`move_relative`/
`collect_each`/`pay_each`/`repairs` — без `send_to_jail`/`get_out_of_jail`,
которые в этом ruleset'е бессмысленны (тюрьмы нет).

## 7. Не входит в эту итерацию

- Свои иконки/3D-модели компаний (используется плейсхолдер-эмблема движка).
- Отдельный анимированный диалог для БИРЖИ (переиспользуется существующая
  контекст-панель HUD + бросок кубика на доске) — при желании можно заменить
  на выделенный экран в следующей итерации, без изменений в `core`.
- Смена символа валюты (`$` захардкожен в UI-хелперах форматирования, не
  завязан на ruleset).

## 8. Проверка (для этапа реализации)

- Юнит-проверка `GameRules.portfolio_tax` (0 полей → 0, N полей → N×ставка).
- Ручной прогон в `solo_test`: остановка на НАЛОГОВОЙ и проезд мимо неё,
  попадание на ОТПУСК (пропуск следующего хода), полный цикл БИРЖИ (3
  броска, оба исхода).
- Проверка, что `classic` ruleset не задет (regression: старая партия и её
  RPC/снапшоты работают как раньше).
