# Известные слабые места и рекомендации

Список составлен по результатам ревью всей кодовой базы (`autoload/`, `core/`,
`render/`, `ui/`, `server/`, `packs/core/`). Отсортировано по важности.

## Критично

### 1. Нет системы контроля версий
В `monopolia/` есть готовые `.gitignore` и `.gitattributes`, но папки `.git`
нет ни в `monopolia/`, ни выше. При 45+ файлах кода и активной разработке
любая неудачная правка невосстановима без ручного бэкапа.

**Рекомендация:** `git init` в `monopolia/` прямо сейчас, первый коммит —
текущее состояние. Убедиться, что `Godot_v4.6.1-stable_mono_win64/` (248 МБ,
дистрибутив редактора) остаётся **вне** репозитория — он лежит на уровень
выше `monopolia/`, так что при `git init` именно внутри `monopolia/` это не
проблема; но стоит проверить `.gitignore`, если репозиторий когда-нибудь
создастся на уровне `D:\MonopolyGodot`.

### 2. Ноль автоматических тестов
`GameRules` (100% статические чистые функции) и `GameController`
(`RefCounted`, не зависит от сцены) — идеальные кандидаты для юнит-тестов,
но их нет вообще. Расчёт ренты, аукцион, сделки, банкротство сейчас
проверяются только вручную через игру.

**Рекомендация:** добавить [GUT](https://github.com/bitwes/Gut) (стандарт
для Godot) и покрыть хотя бы: `GameRules.calculate_rent` (все три типа
собственности + `rent_growth`), полный цикл аукциона, `_validate_trade`,
цепочку `_charge → pending_debt → _try_resolve_debt`.

## Существенно

### 3. `NetworkManager.gd` — растущий God Object
~560 строк, отвечает одновременно за: транспорт, хендшейк, диспетчеризацию
16+ игровых действий, репликацию трёх каналов состояния, зеркало клиента.
Паттерн `if is_server(): local() else: rpc_id(1, remote())` повторяется
почти дословно для каждого действия.

**Рекомендация:** вынести общий диспетчер вида
`_dispatch(rpc_name: String, local_call: Callable)`, чтобы добавление нового
игрового действия не требовало копипасты трёх функций (`request_x`,
`server_request_x`, ветки в `_server_index_action`-подобном роутере).

### 4. Обрыв связи = мгновенное банкротство без реконнекта
`GameController.remove_player()` в фазе `PLAYING` сразу вызывает
`_bankrupt_player`. Короткий сетевой сбой необратимо выбивает игрока из
партии.

**Рекомендация:** ввести окно грейс-периода (например, держать игрока
`connected = false`, пропускать его ходы автопасом, но не банкротить, пока
не истечёт таймаут или другие игроки не проголосуют за кик).

### 5. Нет анти-спам защиты на серверных RPC
Все игровые RPC — `any_peer`, без rate limiting. Для приватного стола с
друзьями не критично, но при выставлении сервера в интернет клиент может
заспамить `server_request_auction_bid`/`server_propose_trade`.

**Рекомендация:** простой троттлинг по `multiplayer.get_remote_sender_id()`
на уровне `NetworkManager`, если планируется публичный сервер.

## Стоит иметь в виду

### 6. Локализация зашита в код
Все строки UI — кириллица прямо в `.gd`-файлах, без `TranslationServer`.
Нормально для проекта на один язык, но усложняет добавление второго языка в
будущем — придётся редактировать каждый экран, а не подключить CSV-перевод.

### 7. Дублирование форматирования денег
`_format_amount`/`_money` (форматирование суммы с разделителями тысяч)
продублированы почти дословно в `Hud.gd`, `TradeDialog.gd`, `AssetDialog.gd`,
`BoardRenderer.gd`. Кандидат на вынос в общий `MoneyFormat`-хелпер рядом с
`ColorUtil`.

### 8. `_effective_sender(fallback)` в `NetworkManager` — заглушка без функции
```gdscript
func _effective_sender(fallback: int) -> int:
    return fallback
```
Выглядит как проверка отправителя, а на деле просто возвращает переданное
значение. Либо реализовать по назначению, либо убрать индирекцию и звать
константу `1` напрямую в местах вызова хостом.

### 9. Целочисленное деление на нечётных `house_cost`
`request_sell_building`: `ledger.credit(peer_id, int(...get("house_cost", 0)) / 2)`
— при нечётном `house_cost` теряется 1 единица валюты (округление вниз).
Сейчас все значения в `cells.json` чётные, так что не проявляется, но это
скрытая хрупкость при добавлении нового контента модом.

## Не проблема, но стоит знать

- `[dotnet]`-секция и mono-сборка Godot используются, но во всём проекте нет
  ни одного `.cs`-файла — вероятно, наследие шаблона проекта, можно
  игнорировать или упростить до обычной (не-mono) сборки движка.
- `physics/3d/physics_engine = Jolt Physics` выбран, но игровой рендер не
  использует физические коллизии (все объекты — чистая геометрия), так что
  выбор физдвижка сейчас не влияет на игру.
