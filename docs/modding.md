# Моддинг и система паков/DLC

Полный гайд для авторов паков (структура `pack.json`, правила иконок/3D-моделей,
рост ренты) уже лежит в **[`mods/README.md`](../mods/README.md)** — это
основной, самодостаточный документ для тех, кто пишет мод. Здесь — как это
устроено изнутри движка, для тех, кто дорабатывает сам `ModLoader`.

## Жизненный цикл загрузки (`ModLoader.load_all`)

```
1. PackRegistry.reset() / AssetLoader.reset() / EventBus.clear_hooks()
2. _discover_manifests()   — сканирует res://packs, res://mods, user://mods
                              (в этом порядке; user:// создаётся автоматически)
3. _resolve_order()        — топологическая сортировка по dependencies+load_after
4. Для каждого пака по порядку:
     _load_pack_content()      — регистрирует JSON из content/*
     _register_scene_overrides()
5. Для каждого пака по порядку (ПОСЛЕ шага 4 для всех паков):
     _run_entry()               — инстанцирует entry-скрипт, вызывает setup(pack_id)
6. _compute_signature()    — SHA-256 по api_version + все манифесты + все
                              файлы контента/ресурсов/entry (не только JSON!)
```

Важно: entry-скрипты запускаются **после** того, как контент всех паков уже
зарегистрирован (шаг 5 отдельно от шага 4) — поэтому пак может в своём
`setup()` безопасно читать контент, который добавила его зависимость, даже
если та зависимость была загружена позже него по каким-то другим причинам.

## `api_version` — почему это жёсткая проверка

`ModLoader.API_VERSION` — константа движка (сейчас `2`). Пак, чей
`api_version` не совпадает, **не загружается вообще** и логируется ошибкой —
это осознанный выбор "громкий отказ вместо тихой порчи состояния". Так же
устроена сетевая проверка: `hello` RPC сравнивает не только `signature`, но
и `server_api` — рассинхрон версии API это отдельная явная причина отказа
(см. [networking.md](networking.md)).

## Сигнатура пака — что именно в неё входит

`_compute_signature` хэширует не только `pack.json`, но и:
- каждый файл в объявленных `content_dirs` (все категории);
- каждый файл в объявленных `resources` (пак-владеющие директории —
  сцены/скрипты/шейдеры/аудио/модели);
- пути `scene_overrides`;
- entry-скрипт.

Это значит: **любое** байтовое изменение в контенте или ресурсах пака меняет
сигнатуру и вызовет отказ подключения при рассинхроне с сервером — даже если
`version` в `pack.json` не поднята. С одной стороны, это надёжно защищает от
молчаливого рассинхрона; с другой — при активной разработке мода сигнатура
будет меняться на каждый чих, так что для локального тестирования это не
проблема, а вот при дистрибуции мода важно не забывать поднимать `version`,
чтобы игроки понимали, что сигнатура изменилась осознанно.

## `resolve_path` — песочница путей ассетов

`AssetLoader.resolve_path(pack_id, rel_path)` отклоняет любой путь,
содержащий `..`, — мод не может сослаться на файл вне своей собственной
папки. Абсолютные `res://`/`user://` пути пропускаются как есть (доверенный
случай — например, встроенный контент). Это единственный явный
security-барьер в системе; **entry-скрипты паков** явно документированы как
"полностью доверенный код" (`mods/README.md`) — то есть песочница защищает
только пути к ассетам, а не выполнение произвольного GDScript. Устанавливать
стоит только моды из доверенных источников.

## Хуки (`EventBus.add_hook` / `run_hook`)

Единственный официальный способ пака вклиниться в игровую логику без
модификации ядра — именованные хуки. Сейчас в `GameController` используется
только один: `"on_land"`, вызываемый для любой клетки, не покрытой встроенными
типами (`property`/`railroad`/`utility`/`tax`/`chance`/`treasury`/`corner_go_to_jail`).
Хук получает `payload` с `peer_id`, `cell`, `board_index`, `jail_index`,
`effects` и должен вернуть (опционально) `Dictionary` с изменённым `effects`
— списком тех же примитивов, что и у карт (`credit`/`debit`/`move_to`/`send_to_jail`,
см. [content-schema.md](content-schema.md#cards--карточки-шанс--казна)).

Приоритет хуков (`priority`, по умолчанию 0) определяет порядок вызова —
выше приоритет выполняется раньше.

## Куда добавлять новые хуки при разработке

Если нужен хук в новой точке ядра (например, "перед списанием ренты"),
паттерн такой же, как `"on_land"`:
1. В `GameController` вызвать `EventBus.run_hook("имя_хука", payload)` в
   нужной точке, где `payload` — обычный `Dictionary` без ссылок на Node/сцену
   (хуки выполняются **только на сервере**, не должны трогать рендер и не
   должны читать баланс других игроков — см. комментарий в `packs/core/scripts/main.gd`).
2. Задокументировать контракт `payload` в `content-schema.md` или отдельном
   разделе — сейчас единственный источник правды по формату `"on_land"` это
   сам код `GameController._resolve_landing`.
